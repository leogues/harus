---
name: harus-commit
description: Organize and create atomic git commits with intelligent change grouping
argument-hint: "[message]"
---

Analyze changes, group them into coherent atomic commits, and create signed commits following repository conventions. This command transforms a messy working directory into a clean, logical commit history.

## Prompt Modes

### Direct Message (Single Commit)

When the user provides a commit message as argument, skip grouping and commit directly:

```bash
# Single commit - message provided
/harus-commit "fix login bug"

# Another example
/harus-commit "add rate limiting to auth endpoints"
```

1. **Skip grouping analysis** - Use provided message directly
2. Stage all changes, ensure proper formatting (50 char subject, etc.)
3. Create signed commit with trailer

```bash
git commit -S \
  -m "fix: fix login bug" \
  --trailer "X-Harus-Ref: 0x1"
```

### No Message (Smart Grouping)

When invoked without a message, follow the full **Commit Process** below:

```bash
# No message - analyzes and groups changes
/harus-commit
```

## Commit Process

### Step 1: Gather Context

Run these commands in parallel to understand the current state:

```bash
# Check staged and unstaged changes
git status

# View ALL changes (staged and unstaged)
git diff
git diff --cached

# View recent commits for style reference
git log --oneline -10
```

### Step 2: Analyze and Group Changes

**Single commit when** all changes are for one coherent feature/fix, user provides a message via argument, or changes are minimal and related.

**Multiple commits when** changes span different concerns (feature + docs + deps), mix of features/fixes/chores, or better git history benefits future archaeology.

**For each changed file, determine:**

1. **Type**: feat, fix, chore, docs, refactor, test, style, perf, ci, build
2. **Scope**: Component or area affected (auth, api, ui, etc.)
3. **Logical group**: What other files belong with this change?

**Grouping heuristics:**

| File Pattern                  | Likely Group                   |
| ----------------------------- | ------------------------------ |
| `*.test.ts`, `*.spec.ts`      | Group with implementation file |
| `package.json`, `*-lock.json` | Dependency changes             |
| `*.md`, `docs/*`              | Documentation                  |
| `*.config.*`, `tsconfig.*`    | Configuration                  |
| Same directory/module         | Often related                  |

**Create a mental (or actual) grouping:**

```
Group 1 (chore): dependencies
  - package.json
  - package-lock.json

Group 2 (feat): auth changes
  - src/auth/oauth.ts
  - src/auth/oauth.test.ts

Group 3 (docs): documentation
  - README.md
```

### Step 3: Determine Commit Order

**Order matters for bisectability:**

1. **Dependencies first** - So subsequent commits can use them
2. **Core changes** - Implementation before consumers
3. **Tests with implementation** - Keep them atomic
4. **Documentation last** - Documents the final state

### Step 4: Present Plan and Confirm

**MANDATORY: Get user confirmation before executing.**

Present the plan visually before asking:

```
Proposed Commit Plan:
─────────────────────
1. chore(deps): update authentication dependencies
   - package.json (modified)
   - package-lock.json (modified)

2. feat(auth): add OAuth2 refresh token support
   - src/auth/oauth.ts (modified)
   - src/auth/oauth.test.ts (modified)

3. docs: update OAuth2 setup guide
   - docs/auth/oauth-setup.md (modified)
```

Then confirm using `AskUserQuestion`. See [ask-commit-plan.md](./assets/ask-commit-plan.md) for the template.

### Step 5: Draft Commit Messages

Follow the repository's existing commit style. If Conventional Commits is used:

```
<type>(<scope>): <subject>

<body - optional>
```

**Guidelines:**

- Subject line: max 50 characters, imperative mood ("add" not "added")
- Body: wrap at 72 characters, explain motivation/context
- The `-m` parameter contains ONLY the commit message — no emoji, no `Co-Authored-By`, no hashtags, no "Generated by", no "X-Harus-Ref" text

### Step 6: Execute Commits (Signed)

**For each commit group, in order:**

1. **Stage only the files for this commit:**

```bash
git add <file1> <file2> ...
```

2. **Create signed commit with trailers:**

```bash
git commit -S \
  -m "<type>(<scope>): <subject>" \
  -m "<body if needed>" \
  --trailer "X-Harus-Ref: 0x1"
```

**`--trailer` is a SEPARATE command-line flag, NEVER inside `-m` quotes:**

```bash
# ❌ WRONG - trailer text inside -m
git commit -m "feat: add feature

X-Harus-Ref: 0x1"

# ✅ CORRECT - --trailer is a separate argument
git commit -S -m "feat: add feature" --trailer "X-Harus-Ref: 0x1"
```

**Required flags:**

- `-S` - GPG sign the commit (REQUIRED for signed commits)
- `--trailer "X-Harus-Ref: 0x1"` - Internal system reference (REQUIRED)

**If GPG signing fails:**

- Check if GPG key is configured: `git config user.signingkey`
- Check if GPG agent is running: `gpg --list-secret-keys`
- If no key configured, proceed without `-S` and inform user

3. **Repeat for each commit group**

### Step 7: Verify Commits

After all commits, verify the result:

```bash
# Show all new commits
git log --oneline -<number_of_commits>

# Verify signatures (if signed)
git log --show-signature -1

# Confirm clean state
git status
```

### Step 8: Offer Push (Optional)

After successful commit, offer to push. See [ask-push.md](./assets/ask-push.md) for the template.

## Important Notes

- **Do not use --no-verify** - Always run pre-commit hooks unless user explicitly requests
- **Trailers are standard** - Git trailers are a recognized convention (like Signed-off-by), machine-readable via `%(trailers)`

## Anti-Rationalization Table

| Rationalization                  | Required Action                              |
| -------------------------------- | -------------------------------------------- |
| "I'll commit everything at once" | **Analyze and group changes first**          |
| "I'll skip GPG signing"          | **Use `-S` flag (skip only if no GPG key)**  |
| "Trailer inside `-m` / HEREDOC"  | **Use `--trailer` as separate flag, always** |
